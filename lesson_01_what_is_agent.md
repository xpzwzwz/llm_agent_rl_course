# 第一课：LLM Agent 是什么

## 1. 从普通聊天到 Agent

普通聊天模型通常只需要回答：

```text
用户：帮我解释一下 PPO。
模型：PPO 是一种强化学习算法...
```

Agent 面对的是任务：

```text
帮我查一下这个库最近的 issue，找出和安装失败有关的问题，并给出修复建议。
```

这类任务不能靠一次回答完成。模型需要搜索、打开页面、筛选信息、比较证据，最后组织结论。

Agent 的基本循环是：

```text
Observation -> Reasoning -> Action -> Result -> Observation -> ...
```

其中 action 可能是搜索网页、打开 URL、读文件、运行测试、调用 API、编辑代码。

## 2. Agent 能力栈

训练 agent 时，不要把“会 function calling”误认为“会 agent”。Function calling 只是基础能力之一。

完整 agent 能力至少包括：

- 工具理解：知道每个工具能做什么、不能做什么。
- 参数生成：能按 schema 正确填写参数。
- 计划分解：把大任务拆成可执行步骤。
- 环境观察：理解工具返回值、网页内容、测试输出、错误日志。
- 自我修正：失败后能换策略，而不是重复同一个动作。
- 状态管理：记住已经做过什么、还缺什么。
- 终止判断：知道什么时候该结束并给出最终答案。

## 3. Agent 和 Workflow 的区别

Workflow 是预先写好的流程：

```text
搜索 -> 摘要 -> 输出
```

Agent 是模型根据环境动态决定下一步：

```text
搜索 -> 发现结果不够 -> 换关键词 -> 打开 GitHub issue -> 读日志 -> 查 release note -> 输出
```

工程里通常会混合使用。生产系统不应该把所有控制权都交给模型。更稳的做法是：

- 用 workflow 固定关键边界和安全检查。
- 用 agent 处理开放式搜索、诊断、选择和修正。
- 用评估器或规则检查最终结果。

## 4. 为什么 Agent 更难训练

单轮回答的损失函数很简单：预测下一个 token。

Agent 训练更难，因为正确性常常取决于整条轨迹：

```text
第 1 步工具选错 -> 第 5 步看似合理 -> 最终失败
```

还有一个问题是延迟奖励。任务最后成功了，不能直接知道中间哪一步贡献最大。任务失败了，也不能直接知道是哪一步导致失败。

所以 agent 训练通常需要混合：

- SFT：学会基本动作格式和工具语义。
- DPO：偏好更好的轨迹。
- RL：用最终任务成功率和过程奖励继续优化。

## 5. 本课练习

拿一个真实任务，手写一条 agent trajectory：

```text
任务：查某个 Python 包安装失败的原因。

Observation: 用户给出错误日志。
Thought: 需要先识别包名和错误类型。
Action: search_web(query="包名 错误关键字")
Result: ...
Thought: 搜索结果显示和 Python 版本有关，需要查官方 issue。
Action: open_url(url="...")
Result: ...
Final: ...
```

重点不是写得像模型内部思考，而是把“看到了什么、做了什么、为什么继续”分清楚。
