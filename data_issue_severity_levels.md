# 数据问题严重级别

这份文档定义数据问题优先级，帮助团队决定先修什么。

## 1. P0：必须立刻停止使用

满足任一条件：

- secret / PII 泄露。
- hidden answer 泄漏到 model-visible 字段。
- train/eval task overlap。
- 大量伪造 observation。
- reward/verifier 被模型篡改。
- 越权或危险动作被当成 chosen/SFT。

处理：

- 暂停该批数据。
- 从训练池移除。
- 追踪影响过的模型版本。
- 重新生成或清洗。

## 2. P1：训练前必须修

例子：

- DPO prompt 不一致。
- chosen/rejected 标签大规模冲突。
- action 格式多套混用。
- tool schema 与线上不一致。
- truncated pair rate 高。
- synthetic 模板化严重。

处理：

- 修转换脚本。
- 重新标注。
- 降权或剔除。
- 重新跑质检。

## 3. P2：可以低权重使用，但要监控

例子：

- 轨迹略冗余。
- tool result 摘要有信息损失。
- reward gap 偏小。
- 失败类型缺少细分类。
- 少量格式错误。

处理：

- 降权。
- 抽查。
- 在训练报告中标记风险。

## 4. P3：记录即可

例子：

- 文案风格不统一。
- 非关键 metadata 缺失。
- 少量重复 prompt。
- final 表达不够简洁但任务成功。

处理：

- 记录。
- 后续版本优化。

## 5. 优先级判断表

```text
是否涉及隐私/安全？是 -> P0
是否污染评估？是 -> P0
是否会教模型错误行为？是 -> P0/P1
是否影响主要训练信号？是 -> P1
是否只是降低效率或增加噪声？是 -> P2
是否只是文档/风格问题？是 -> P3
```

## 6. 修复顺序

优先级：

```text
P0 安全/泄漏/评估污染
-> P1 格式/标签/协议问题
-> P1 数据阶段错配
-> P2 噪声和冗余
-> P3 文档和风格
```

不要在 P0 未清除时讨论调超参。
